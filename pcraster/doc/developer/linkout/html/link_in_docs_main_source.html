37  \mainpage LinkIn: API to add functionality to the PCRaster Model Engine
38 
39 
40  \section introduction Introduction
41 This document documents the Application Programming Interface (API) for
42 adding new functionality to the pcrcalc modelling engine.
43 
44 
45 The API enables programmers to write a shared library
46 in their preferred programming language and expose the
47 functionality to the PCRaster model engine (pcrcalc or PCRasterPython).
48 Functionality can be exposed as simple (stateless) functions with a number of input and output arguments or statefull objects with methods changing the object state. State full objects allows to link your model script with:
49  - other complex pieces of software such as other models (See Linking external components to a spatio-temporal modelling framework: Coupling MODFLOW and PCRaster: http://dx.doi.org/10.1016/j.envsoft.2009.02.018).
50  - use your own data structures beyond raster maps and nonspatials inside the object.
51 
52 
53 The following PCRaster script uses a library to implement a function that generates
54 a boolean checker board true/false pattern:
55 
56 \verbatim
57 areamap area.map;
58 checkerBoard.map = example1::checkerBoard();
59 \endverbatim
60 
61 When executed, pcrcalc expects to find a library named example1 (example1.dll or libexample1.so). That
62 library will implement the function checkerBoard. The library should be
63 accompanied by a manifest XML file with the same name and the extension
64 .xml (example1.xml ). The XML file describes which functions and classes are present in the library.
65 
66 The API allows for two use modes:
67 - simple mode: implement a single function in one library requiring minimal knowledge of XML.
68 - advanced mode: anything beyond the simple mode.
69 
70 \section StatusQuo Status Quo
71 
72 The API presented in this document is a new API to be used with the 2006 and more recent versions of pcrcalc (not oldcalc.exe) and is not compatible with the C-API used in oldcalc.exe ( http://pcraster.geog.uu.nl/pcrcalcapi.html).
73 
74 The new LinkIn API is very much work in progress. Therefor this documentation and the code samples are not yet part of the PCRaster distribution.
75 
76 As a final product a layered architecture is envisioned:
77  - Protocol layer
78  - optional RPC layer to allow distributed computing
79  - Language Runtime layer
80 
81 On the lowest level there is the LinkIn Protocol layer that only deals with memory transfer of data and an XML string that describes the request to the library called. The protocol assumes very little except a handful of C linkable routines to be present in the library created.
82 
83 The LinkIn Language Runtime Layer is a programming language (C, C++, Fortran, etc.) specific library that should provide:
84 
85  - Wrappers around the Protocol routines with embedded XML processing. When using the Language Runtime layer no XML processing should be needed by the library developer.
86  - Convenience wrapper objects/functions for maps as 2D array in either Row or Column-major order.
87  - NoData/Missing Value utilities
88  - error messaging or exceptions in the way described (best practice) by the language.
89 
90 Currently only the Protocol layer is implemented and successfully deployed in complex situations by the PCRaster development team self.

100 \section SimpleMode Simple mode
101 
102 The checkerBoard function above is the most simple and can be found in the linkinexamples/example1 directory. For the introduction of simple mode the linkinexamples/example2 directory is used with
103 the function addSpatialToNonSpatial
104 
105 Simple mode can implement a single function f in one library of the form:
106 
107  r = f(a1,a2,...,an)
108 
109 With result r and n (0 or more) input arguments a<i>i</i>. Both r and a<i>i</i> can only have a single
110 datatype (e.g. boolean or scalar) and should be either Spatial or NonSpatial (not Either).
111 For the implementation of the function addSpatialToNonSpatial we would write an XML manifest (example2.xml) like:
112 
113 \verbatim
114 <linkInLibraryManifest ... >
115 <function>
116  <name>addSpatialToNonSpatial</name>
117  <result>
118  <dataType><scalar/></dataType>
119  <spatialType>Spatial</spatialType>
120  </result>
121  <argument>
122  <dataType><scalar/></dataType>
123  <spatialType>Spatial</spatialType>
124  </argument>
125  <argument>
126  <dataType><scalar/></dataType>
127  <spatialType>NonSpatial</spatialType>
128  </argument>
129 </function>
130 </linkInLibraryManifest>
131 \endverbatim
132 
133 This manifest tells pcrcalc that the function addSpatialToNonSpatial will return a spatial of scalar
134 type and takes two input arguments: the first one is a scalar spatial and the second one a scalar nonspatial. To create your own manifests, simply copy the example2.xml and:
135 - change addSpatialToNonSpatial to the name of your function,
136 - for each result copy a result element,
137 - for each argument copy an argument element,
138 - set each dataType element to one of the following values: scalar, directional, boolean, nominal, ordinal or ldd,
139 - set each spatialType element to Spatial or NonSpatial.
140 
141 In case of typing errors the PCRaster modelling engine will point out the position of the error by line and column (char) number.
142 
143 To implement this function in the library the library should have a function named pcr_LinkInExecute().
144 This function will pass an LinkInTransferArray[] containing memory space for both the result and the 2 arguments. The pcr_LinkInExecute() function of example 2 contains the code needed for the addSpatialToNonSpatial function. Since all dataTypes and spatialTypes are known in the XML manifest, the example2.c code can simply use the correct types known at compile time.
145 
146 In simple mode, no XML processing for the first argument of pcr_LinkInExecute() is needed. This argument does however encode the nrRows and nrCols that is needed to determine the size of the spatial buffers. But this information is also passed to the library, if the library implements the function pcr_LinkInRunContext().

151 \section AdvancedModeIntro Advanced Mode Introduction
152 
153 More advanced libraries are possible by passing information encoded in XML back and forth between the engine and the library. The XML messages are documented by the schema PCRaster.xsd.
154 The XML elements used in the LinkIn communication scheme are defined in PCRaster.xsd. In this document they are identified by the prefix pcrxml::.
155 
156 The communication scheme is as follows:
157 - on start up the XML Manifest is read (pcrxml::linkInLibraryManifest),
158 - when parsing the script the engine asks type information from the library by calling pcr_LinkInCheck(),
159 - when executing the engine passes type information in the first argument of
160 pcr_LinkInExecute().
161 
162 This scheme allows for implementation of more powerful functions than simple mode. For example the pcrcalc cover operator since it allows:
163 
164 - multiple data types as dataType,
165 - pcrxml::SpatialType Either as spatialType,
166 - variable number of input arguments.
167 
168 The LinkIn library uses the Object-Oriented paradigm that defines classes with constructors and methods. A script using objects (e.g. instance of a class) may look like this:
169 
170 \verbatim
171 
172 # a class constructor
173 object obj1 = aLinkInLibrary::Class1("initObj1.txt", ...);
174 # a class constructor
175 object obj2 = aLinkInLibrary::Class1("initObj2.txt", ...);
176 
177 # a method call
178 value = obj1::method1(dem)*3;
179 
180 \endverbatim
181 
182 The example above demonstrates a number of features of the LinkIn libraries:
183 - the keyword object defines a constructor.
184 - all functions, constructors and methods may have a first optional string argument that will be passed in pcrxml::stringArgument.
185 - a method call is identified by the object name plus the method name.
186 
187 \section AdvancedModeImpl Advanced Mode Implementation
188 
189 A C++ example of advanced mode is provided in the exampleCpp directory.
190 This example uses xsd, http://codesynthesis.com/projects/xsd/, an XML-C++ data binding tool to do the XML processing.
191 
192 Since the API consists only of a few C functions, a LinkIn library should implement all logic.
193 The pcr_LinkInCheck() and pcr_LinkInExecute() functions are merely dispatchers that will call the correct functions and methods. Also note that the construction and destruction of objects is the responsibility of the library. Objects are only identified by name.
194 
195 As one can see from this example, a lot of code is involved for doing the dispatch to the correct function and method. For that we envision the Language Runtime Layer that will hide all the XML details from the developer.
196 
197 \section BuildingAndInstalling Build and Install Libraries
198 
199 You can use any programming language that:
200  - can create a shared library (dll in Windows, shared .so library on linux)
201  - allows to define functions with C style linkage
202 
203 The library must always contain the function pcr_LinkInExecute().
204 
205 The code libraries plus their XML manifest can be installed in the same place as pcrcalc, or one can install the libraries and manifests in a different place based on the PATH (Windows) or LD_LIBRARY_PATH (linux) setting.
